# 手写笔通信与芯片模式切换逆向记录

## 1. 文档范围

本记录基于 Ghidra 对以下原厂模块的静态逆向：

- `THP_Service.dll`（服务层，USB/事件分发）
- `himax_thp_drv.dll`（触控芯片命令下发）
- `TSACore.dll`（频移策略状态机，辅助确认）

目标是回答两件事：

1. 手写笔（蓝牙相关）在原厂驱动中的实际交互路径是什么。
2. 120Hz/240Hz 自动切换（混合模式）命令如何下发，具体命令是什么。

## 2. 总体结论

### 2.1 手写笔"蓝牙交互"结论

- Windows 侧没有看到直接 BLE API（如 `Bluetooth*`/`Bth*`）调用。
- `THP_Service.dll` 通过 USB 设备句柄与一个 MCU 设备通信，MCU 再与笔做蓝牙交互。
- 因此主机侧形态是：`THP_Service <-> MCU(USB私有协议) <-> Pen(BT)`。

### 2.2 芯片模式切换结论

- `THP_Service!ThpFuncStart` 只负责服务启动编排，不直接给 Himax 芯片下混合模式命令。
- 真正切换命令在 `himax_thp_drv.dll` 的 `hx_send_command` 路径。
- 开启自动频移核心命令是 `cmd_id=0x0D`（`thp_afe_enable_freq_shift`）。

### 2.3 VHF 回报逆向结论

- `THP_Service!ServiceInterfaceInit0`
  - `pFuncReportCoordinates = VhfWriteFingerFile`
  - `pFuncInjectPenData = VHF_InjectPenData`
- `THP_Service!VhfWriteFingerFile`
  - `WriteFile(len=0x20)`，触摸包固定 32 字节。
- `THP_Service!VHF_InjectPenData`
  - `WriteFile(len=0x0D)`，手写笔包固定 13 字节。
- `THP_Service!VHF_FindAndOpen_Wrapper`
  - 设备接口 GUID：`{59819B74-F102-469A-9009-3CAF35FD4686}`。

## 3. 手写笔通信（THP_Service）

### 3.1 关键函数与职责

- `Usb_Start` @ `0x1800116e0`
  - 创建读线程和处理线程。
  - 启动后主动发送两条初始化查询命令（`0x7101`、`0x7701`）。
- `AsynchReadThreadProc` @ `0x18000e1d0`
  - `ReadFile(..., 0x40)` 循环收包。
  - 收到有效包后写入环形队列（深度 `0x400`）。
- `USB_AsynchProcThreadProc` @ `0x18000cf40`
  - 从环形队列取包，按事件码分发（`0x70~0x7F` 等）。
  - 更新笔状态位并回 ACK（`BtPen_SendEventAck`）。
- `BtPen_FindAndOpen` @ `0x18000f270`
  - 使用 `SetupDi* + CreateFileW` 打开 MCU 设备接口句柄。
- `BtPen_SendPacket` @ `0x18000f500`
  - 构造并 `WriteFile` 发送 MCU 包。
- `BtPen_GetReportInfo` @ `0x18000e960`
  - 处理上层主动请求（版本/初始化参数/匹配信息）。
- `BtPen_HandleInitParamEvent` @ `0x18000f660`
  - 解析 INIT_PARAM_EVENT 参数并下发 `0x7D01`。
- `BtPen_SendEventAck` @ `0x180011d20`
  - 对 MCU 上报事件回 ACK（`0x8001`）。

### 3.2 USB 私有协议发送格式（服务层看到的形态）

发送统一走 `BtPen_SendPacket`：

- 先准备 8 字节报文头；
- 函数内部会固定 `header[6] = 0x11`；
- 业务 payload 从偏移 `+8` 开始拷贝；
- 实际写长度：`payload_len + 8`。

观察到的典型头部字节（小端）：

- `0x7101` 查询：`07 00 02 00 01 71 11 00`
- `0x7701` 查询：`07 00 02 00 01 77 11 00`
- `0x7D01` INIT 参数下发：`07 01 02 00 01 7D 11 20`
- `0x7E01` 匹配信息回复：`07 01 02 00 01 7E 11 01`
- `0x8001` 事件 ACK：`07 01 02 00 01 80 11 20`

说明：

- `cmd_id` 位于头部第 4~5 字节（小端）。
- 头第 7 字节在不同命令下语义不完全一致（不能简单等同 payload 长度）。

### 3.3 事件分发表（上行 MCU->Service）

`USB_AsynchProcThreadProc` 中已确认的主要事件：

- `0x70` `PEN_AC_STATUS`
- `0x71` `PEN_CONN_STATUS`
- `0x72` `PEN_CUR_STATUS`
- `0x73` `PEN_TYPE_INFO`
- `0x74` `PEN_ROATE_ANGLE`
- `0x75` `PEN_TOUCH_MODE`
- `0x76` `PEN_GLOBAL_PREVENT_MODE`
- `0x78` `PEN_HOLSTER`
- `0x79` `PEN_FREQ_JUMP`
- `0x7C` `PEN_GLOBAL_ANNOTATION`
- `0x7F` `ERASER_TOGGLE`
- 以及 `DEV_PAIR_STATUS / PEN_DOCK_STATUS / PEN_UPDATE_STATUS / PEN_KEY_FUNC_GET / ...`

### 3.4 ACK 机制

事件处理后通常调用 `BtPen_SendEventAck(code)`，ACK 映射：

- `0x70 -> ack 0`
- `0x71 -> ack 1`
- `0x72 -> ack 2`
- `0x73 -> ack 13`
- `0x74 -> ack 3`
- `0x75 -> ack 4`
- `0x76 -> ack 5`
- `0x77 -> ack 6`
- `0x78 -> ack 7`
- `0x79 -> ack 8`
- `0x7B -> ack 10`
- `0x7C -> ack 12`
- `0x7F -> ack 9`
- `0x2F -> ack 11`

### 3.5 与系统/上层的桥接

- `ServiceInterfaceInit2` 注册 USB 与笔相关回调：
  - `Usb_Start` / `Usb_Stop`
  - `GetPenStatusFlags`
  - `HandlePenStatusCode`
  - `BtPen_GetReportInfo`
- `HandlePenStatusCode` 把状态转成高层动作字符串：
  - `-connect / -disconnect / -erase / -doubleclick / -start ...`
- 动作经 `PenkitPipe_SendMessage`（原 `FUN_1800063e0`）发往本地命名管道：
  - `\\.\Pipe\2F4B70D9618044F1B64B3D25E5FD3AF1`

另有 `CommandThread` 在 `127.0.0.1:9999` 起本地 TCP 控制口（进程间控制），不属于笔蓝牙链路本身。

## 4. 芯片模式切换（himax_thp_drv）

### 4.1 核心发送函数

- `hx_send_command` @ `0x1800119a0`

命令帧固定 16 字节：

- `buf[0]=0xA8`
- `buf[1]=0x8A`
- `buf[2]=cmd_id`
- `buf[3]=0x00`
- `buf[4]=cmd_val`
- `buf[14..15]=checksum`

发送步骤：

1. 计算 16-bit 校验并填入 `buf[14..15]`。
2. 写入命令槽：`addr = 0x10007550 + slot*0x10`。
3. 再写触发头（4字节）：`A8 8A cmd_id 00`。
4. 回读同地址 16 字节校验。
5. `slot = (slot + 1) % 5`。

### 4.2 与混合模式相关的命令字典

以下均由 `hx_send_command` 下发：

- `thp_afe_enable_freq_shift`：`cmd_id=0x0D, cmd_val=0x00`
- `thp_afe_disable_freq_shift`：`cmd_id=0x02, cmd_val=0x00`
- `thp_afe_clear_status`：`cmd_id=0x06, cmd_val=status`
  - `status=4` 清 `freq-shift done`
  - `status=8` 清 `calibration done`
- `thp_afe_force_to_freq_point`：`cmd_id=0x0C, cmd_val=freq_index`
- `thp_afe_force_to_scan_rate`：`cmd_id=0x0E, cmd_val=0x00`
- 相关：`thp_afe_enter_idle`=`0x0A`，`thp_afe_force_exit_idle`=`0x0B`，`thp_afe_start_calibration`=`0x01`

### 4.3 典型 16 字节示例帧（含校验）

- Enable Freq Shift (`0x0D,0x00`)
  - `A8 8A 0D 00 00 00 00 00 00 00 00 00 00 00 4B 75`
- Disable Freq Shift (`0x02,0x00`)
  - `A8 8A 02 00 00 00 00 00 00 00 00 00 00 00 56 75`
- Clear FreqShiftDone (`0x06,0x04`)
  - `A8 8A 06 00 04 00 00 00 00 00 00 00 00 00 4E 75`
- Force To Scan Rate (`0x0E,0x00`)
  - `A8 8A 0E 00 00 00 00 00 00 00 00 00 00 00 4A 75`
- Force To Freq Point 0 (`0x0C,0x00`)
  - `A8 8A 0C 00 00 00 00 00 00 00 00 00 00 00 4C 75`

### 4.4 与 120/240Hz 自动切换的关系

- 从驱动调用关系看，"自动切换"不是直接下发 `120/240` 数值，而是启用频移能力：
  - 入口：`thp_afe_enable_freq_shift(0x0D)`
  - 后续由 AFE/策略状态机根据笔状态与噪声/频点状态进行切换。
- `THP_Service` 中 `PEN_FREQ_JUMP(0x79)` 分支仅日志+ACK，不直接向 Himax 下发频率命令。

## 5. 启动到切频的端到端时序（简化）

1. `ThpFuncStart` 启动 VHF/USB/SPI 子系统。
2. `Usb_Start` 拉起读线程和处理线程，先发 `0x7101/0x7701` 查询。
3. MCU 回报笔状态事件（`0x70~0x7F`），服务层更新状态并回 `0x8001 ACK`。
4. AFE 层（`himax_thp_drv`）在已启动状态下调用 `0x0D` 开启频移。
5. 运行时根据状态机可触发 `0x0C/0x0E/0x06` 等辅助命令。
