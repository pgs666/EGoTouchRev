# EGoTouchRev 架构现状梳理与诊断报告

## 1. 为什么感觉“特别乱”？（宏观诊断）

您当前的代码库正处于两代架构的**交汇期**：
- **旧版实现基座**：`App/RuntimeOrchestrator` 目前是一个极其庞大的“上帝类（God Class）”。它不仅负责所有 3 个核心线程的调度，还**直接耦合了硬件操作**（持有 `HimaxChip` 实例）、笔迹协议桥接（`PenCommandApi`）、以及系统接口注入（VHF/HID）。
- **新版架构蓝图（plan.md）**：您在 [plan.md](file:///d:/source/repos/EGoTouchRev-rebuild/plan.md) 中设计了极其优雅的“[Device](file:///d:/source/repos/EGoTouchRev-rebuild/App/include/RuntimeOrchestrator.h#33-35) 自维持采集”、“`Host` 事件上报”、“[Control](file:///d:/source/repos/EGoTouchRev-rebuild/Control/include/ControlRuntime.h#20-21) 异步命令调度”分层模式。
- **现状撕裂**：您已经按照新版蓝图建好了 [Control](file:///d:/source/repos/EGoTouchRev-rebuild/Control/include/ControlRuntime.h#20-21) 模块（写好了强大的 [ControlRuntime](file:///d:/source/repos/EGoTouchRev-rebuild/Control/include/ControlRuntime.h#20-21) 队列机制），也建好了 `Host` 模块（写好了系统事件监听 [SystemStateMonitor](file:///d:/source/repos/EGoTouchRev-rebuild/Host/include/SystemStateMonitor.h#14-48)），**但是这套优雅的新系统还没有被 `App/RuntimeOrchestrator` 接纳**。[App](file:///d:/source/repos/EGoTouchRev-rebuild/App/source/RuntimeOrchestrator.cpp#367-376) 仍在使用一套老旧的、直接调 `Himax::Chip` 接口的方式在运行，这导致代码在逻辑上分成了两套控制流，这是您感到混乱的根本原因。

## 2. 各模块开发现状剖析

根据我的全局扫描，目前 6 大模块的实际完成度如下：

### ✅ Engine (触控解算引擎) - 完成度很高，职责清晰
这里是目前实现最好、最清晰的地方。
- 采用管道模式 ([FramePipeline](file:///d:/source/repos/EGoTouchRev-rebuild/Engine/include/FramePipeline.h#8-25)) 串联了 7 个以上的处理器（`MasterFrameParser` -> `BaselineSubtraction` -> `CMF` -> `GridIIR` -> `FeatureExtractor` -> `StylusProcessor` -> `TouchTracker`）。
- **状态：** 代码独立性极高，完全解耦。不需要大动。

### 🔄 Device (硬件抽象层) - 处于半成品状态
- `himax/HimaxChip`：目前只提供了同步的 API (`GetFrame`, `switch_afe_mode` 等)。
- `btmcu/PenCommandApi`：包含了通过蓝牙/USB接收触控笔状态的逻辑。
- **核心缺失（plan.md 明确指出）：** 缺少 `DeviceRuntime`（状态机+线程循环）。目前 [Chip](file:///d:/source/repos/EGoTouchRev-rebuild/Device/include/himax/HimaxChip.h#126-127) 的轮询仍由上层的 `App/RuntimeOrchestrator` 越俎代庖。

### 🚧 Host (系统集成层) - 已写好，但未深度联动
- [SystemStateMonitor](file:///d:/source/repos/EGoTouchRev-rebuild/Host/include/SystemStateMonitor.h#14-48)：已经完美实现了 Windows 事件监听（如熄屏、亮屏）。
- **现状：** 在旧的 `RuntimeOrchestrator::SystemStateThreadFunc` 中，还是一个每秒睡眠的长循环（TODO 待接入）。而在新的 `ControlServiceMain` 中已经正确装配，但 Service 主逻辑还未替换掉老的 App 主逻辑。

### 🏗️ Control (控制与策略服务) - 新鲜出炉，架构超前
- 这是 [plan.md](file:///d:/source/repos/EGoTouchRev-rebuild/plan.md) 中设计的亮点。包含 [ControlRuntime](file:///d:/source/repos/EGoTouchRev-rebuild/Control/include/ControlRuntime.h#20-21)（命令队列、优先级、消抖、历史记录）以及跨进程用的 [ControlPipeServer](file:///d:/source/repos/EGoTouchRev-rebuild/Control/include/ControlPipeServer.h#11-35)。
- **现状：** 目前作为 `EGoTouchControlService` 存在，但它内部只是个框架，它**尚未拿到 [Device](file:///d:/source/repos/EGoTouchRev-rebuild/App/include/RuntimeOrchestrator.h#33-35) 的控制权**，命令执行下去并没有真正去调用硬件 API。

### ⚠️ App (入口与调度) - 沉重的历史包袱中心
- [RuntimeOrchestrator](file:///d:/source/repos/EGoTouchRev-rebuild/App/include/RuntimeOrchestrator.h#25-144) 极其臃肿，承担了不该承担的职责：
  - 手写硬件轮询 ([AcquisitionThreadFunc](file:///d:/source/repos/EGoTouchRev-rebuild/App/source/RuntimeOrchestrator.cpp#198-226) 强耦合 `m_device->GetFrame()`)
  - 手搓同步/时序强绑定的业务（由于 `DeviceRuntime` 不存在，把本该下沉的频移同步逻辑 [HandleAutoAfeFreqShiftSync](file:///d:/source/repos/EGoTouchRev-rebuild/App/source/RuntimeOrchestrator.cpp#638-713) 放在了这里）。
  - VHF 虚拟注入也耦合在这里进行 ([EnsureVhfDeviceOpenLocked](file:///d:/source/repos/EGoTouchRev-rebuild/App/source/RuntimeOrchestrator.cpp#522-576), [DispatchVhfReports](file:///d:/source/repos/EGoTouchRev-rebuild/App/source/RuntimeOrchestrator.cpp#497-521))。
- 此外，作为调试利器的 [DiagnosticsWorkbench](file:///d:/source/repos/EGoTouchRev-rebuild/App/include/DiagnosticsWorkbench.h#8-60)（GUI）严重依赖 [RuntimeOrchestrator](file:///d:/source/repos/EGoTouchRev-rebuild/App/include/RuntimeOrchestrator.h#25-144) 这个臃肿的单例。

## 3. 面向 V1.1 架构的演进路线规划（您的下一步）

建议您**停下增加新功能的脚步，先拔掉这根刺**，专心完成**架构切换的重构任务**。

按照您自己的 [plan.md](file:///d:/source/repos/EGoTouchRev-rebuild/plan.md) 中的“阶段实施计划”，我为您提炼出接下来应该做的 3 个核心重构步骤（需要一步步来，一定不能急）：

### 🚨 步骤一：创建 DeviceRuntime （下放采集循环）
**当前痛点**：`HimaxChip` 只是个操作集合，轮询却交给了 [App](file:///d:/source/repos/EGoTouchRev-rebuild/App/source/RuntimeOrchestrator.cpp#367-376)，导致通信和硬件控制逻辑分散。
**行动指南**：
1. 在 `Device/source` 和 `Device/include` 下，新建 `DeviceRuntime.h/cpp`。
2. 将 `App/RuntimeOrchestrator` 里的 `m_acquisitionThread` 逻辑**剪切**到 `DeviceRuntime` 中。
3. `DeviceRuntime` 内部创建一个数据输出管道（RingBuffer），把采集到的原始帧推出来。
4. 让 `DeviceRuntime` 接管 `HimaxChip` 实例的生命周期。外部（如 `App`）不再能直接 `GetDevice()` 拿到硬件句柄，只能拿到 `DeviceRuntime`。

### 🚨 步骤二：连接 ControlRuntime 与 DeviceRuntime（命令下行）
**当前痛点**：UI 直接调硬件、笔协议调硬件，存在并发危险且无法做策略缓冲。
**行动指南**：
1. 补全 `ControlRuntime::Execute()` 函数中的 switch-case。
2. 当 `ControlRuntime` 收到 `ControlCommandKind::StartStreaming` 时，去调用 `DeviceRuntime::StartStreaming()`。
3. 当 `ControlRuntime` 收到 `ControlCommandKind::EnterIdle` 时，去调用 `DeviceRuntime::EnterIdle()`。
4. 确保所有对硬件操作的行为（由用户点击 UI 等引起）均转为 `SubmitCommand`，彻底掐断所有模块直接调用 `HimaxChip` 的情况。

### 🚨 步骤三：解耦 RuntimeOrchestrator (瘦身)
**当前痛点**：这个大脑太重了，且掺杂了系统监控和虚拟输入。
**行动指南**：
1. `RuntimeOrchestrator` 退化为一个纯粹的 **Pipeline与VHF装配车间**。
2. 剥除 `AcquisitionThread`，它是 `DeviceRuntime` 生产原始热力图流的中转站，交由 Engine 运算。
3. 把 `SystemMonitor` 独立出来，连线其产出的 Event 给到 `ControlRuntime`（作为自动熄屏降帧的输入）。
4. （可选）把 VHF 注入逻辑（`VhfInjector`）像您最初计划那样移到 `Host/` 里。

## 4. 灵魂拷问：当前我该用纯服务态还是双端GUI态？

在您的 CMake 中，我看到同时存在 `EGoTouchApp` (带 GUI 的独立可执行文件) 和 `EGoTouchControlService` (C/S 架构，通过 Pipe 通信的服务)。

这导致您的入口非常分裂。如果您想让代码不乱，**请在此刻做出决断**：

- **方案A（推荐调试期使用）：融合单体架构（Monolithic App）**
  暂时放弃 `ControlPipeClient/Server` 这个跨进程宏图。把 `ControlRuntime` 和 `DeviceRuntime` 直接以对象聚合的方式组装在 `App/main.cpp` 中。同时通过 ImGui (也就是现有的 `DiagnosticsWorkbench`) 直接观察。这样调试极为方便，且没有进程间通信的复杂性。

- **方案B（生产部署环境）：服务/调试器分离架构（C/S）**
  把所有硬件轮询 (`DeviceRuntime`) + 管线计算 (`Engine`) 丢进服务级后台 `EGoTouchControlService` 中运行。
  GUI `DiagnosticsWorkbench` 则蜕变成一个纯粹的客户端程序，它只通过 `ControlPipeClient` 去轮询拿到后台计算好的最新热帧用于显示。此方案对进程间通信的要求极高（需要把几十KB一帧的热力图通过共享内存或高频管道塞给 GUI），实现成本较高，但最符合最终发布给客户的“免打扰后台驱动”。

### 我的建议：
在算法和硬件逻辑还未 100% 稳如泰山前，**坚决使用方案 A**。不要把本来可以用同一个内存槽看的指针搞成跨进程调用，这会极大地增加您的心智负担。
