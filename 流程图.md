# DeviceRuntime 控制图

下面的流程图基于当前仓库现状与建议演进方向整理，目标是把 `Himax::Chip` 的访问统一收敛到 `DeviceRuntime`，由它负责命令执行、状态维持、采集与恢复。

## 1. 总体控制链路

```mermaid
flowchart TD
    UI[Diagnostics UI<br/>手动按钮/调试命令]
    HOST[Host / SystemStateMonitor<br/>亮屏 熄屏 Lid 事件]
    ENGINE[Engine 策略闭环<br/>FreqShift Idle 建议]

    POLICY[ControlRuntime / Policy Layer<br/>去重 优先级 策略翻译 限流]
    RUNTIME[DeviceRuntime<br/>命令队列 + 条件变量 + Worker线程]
    STATE[CurrentState 状态机<br/>Uninit Init Ready Streaming Idle Recovering Error]
    CHIP[Himax::Chip<br/>Init Deinit GetFrame switch_afe_mode check_bus]
    BUFFER[Raw Frame Buffer<br/>覆盖式 RingBuffer]
    METRICS[Runtime Metrics<br/>timeout error fps recovery]
    APP[RuntimeOrchestrator / Engine<br/>算法处理 VHF GUI取帧]

    UI --> POLICY
    HOST --> POLICY
    ENGINE --> POLICY
    POLICY -->|SubmitCommand| RUNTIME
    RUNTIME --> STATE
    STATE --> CHIP
    CHIP --> BUFFER
    CHIP --> METRICS
    BUFFER --> APP
```

## 2. DeviceRuntime 状态机

```mermaid
stateDiagram-v2
    [*] --> Uninitialized

    Uninitialized --> Initializing: Init
    Initializing --> Ready: Chip::Init 成功
    Initializing --> Error: Init失败

    Ready --> Streaming: StartStreaming
    Ready --> Stopping: Deinit / Shutdown

    Streaming --> Idle: EnterIdle
    Idle --> Streaming: TouchWakeup / ExitIdle

    Streaming --> Recovering: 连续超时 / 通信异常
    Idle --> Recovering: 通信异常
    Recovering --> Streaming: 恢复成功且需继续采集
    Recovering --> Ready: 恢复成功但未进入采集
    Recovering --> Error: 超过恢复上限 / BusDead

    Streaming --> Stopping: StopStreaming / Deinit / Shutdown
    Idle --> Stopping: Deinit / Shutdown
    Error --> Stopping: Shutdown / 人工停止
    Stopping --> Uninitialized: 收尾完成
```

## 3. Worker 主循环

```mermaid
flowchart TD
    START([Worker线程启动]) --> WAIT[等待命令或周期唤醒]
    WAIT --> CMD{收到命令?}

    CMD -->|是| EXEC[执行命令<br/>更新 currentState<br/>调用 Chip 接口]
    EXEC --> WAIT

    CMD -->|否| SWITCH{currentState}

    SWITCH -->|Uninitialized| IDLE0[空闲等待 Init]
    IDLE0 --> WAIT

    SWITCH -->|Initializing| DOINIT[执行 Chip::Init]
    DOINIT --> WAIT

    SWITCH -->|Ready| IDLE1[等待 StartStreaming / EnterIdle / Deinit]
    IDLE1 --> WAIT

    SWITCH -->|Streaming| GETFRAME[GetFrame -> 发布帧<br/>更新 fps timeout error 统计]
    GETFRAME --> CHECK1{成功?}
    CHECK1 -->|是| WAIT
    CHECK1 -->|否| REC1[累计异常并请求进入 Recovering]
    REC1 --> WAIT

    SWITCH -->|Idle| IDLEFRAME[保持 Idle 读策略<br/>短等待或低频读帧]
    IDLEFRAME --> CHECK2{需要唤醒/异常?}
    CHECK2 -->|否| WAIT
    CHECK2 -->|是| WAIT

    SWITCH -->|Recovering| RECOVER[轻恢复 -> 中恢复 -> 重恢复<br/>check_bus / IntClose+Open / Deinit+Init]
    RECOVER --> CHECK3{恢复成功?}
    CHECK3 -->|是| WAIT
    CHECK3 -->|否| ERR[进入 Error 或继续退避重试]
    ERR --> WAIT

    SWITCH -->|Error| HOLD[等待人工命令或策略重启]
    HOLD --> WAIT

    SWITCH -->|Stopping| STOPPING[停止采集 关闭句柄 释放资源]
    STOPPING --> END([线程退出或回到 Uninitialized])
```

## 4. 设计要点

- `Himax::Chip` 只允许 `DeviceRuntime` 独占访问。
- 外部模块提交的是“命令”，不是直接修改设备内部状态。
- `currentState` 只允许 worker 线程修改，避免并发竞态。
- 恢复流程必须是状态机的一部分，不能只是错误后直接返回。
